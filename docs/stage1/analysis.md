# 方案 A 的能力边界与示例

## 1. 方案定位

当前选择的方案 A 是：**函数调用图 + 参数/返回值映射**。

Vode 将不再尝试把所有模块、函数、算子、基础表达式统一成一个全知全能的执行图系统，而是先收缩为一个更务实的目标：

> 以函数调用边界为核心，尽量高保真地记录函数之间的数据输入、输出与流向；对于 `torch.nn` 相关对象，增加专项支持，使得神经网络场景下的展示效果更强。

这个方案的关键点是：

1. 主体使用 Python 运行时 trace 能力，跟踪函数 `call` / `return`
2. 在函数边界读取当前 frame，获取参数名、参数对象、返回值对象
3. 用对象身份和事件索引建立“谁生产了某个对象、谁又消费了它”的映射
4. 对 `torch.nn.Module`、parameter、buffer、tensor 元信息做专项增强

---

## 2. 当前方案能够实现的功能

## 2.1 零侵入启动

用户原则上无需修改自己的业务代码，只需要将运行方式从：

```bash
python model/train.py
```

改成：

```bash
vode python model/train.py
```

Vode 在启动时注入 trace 逻辑并记录执行事件。

这属于“对用户源码零侵入”，但并不意味着“对运行时完全无插桩”。

---

## 2.2 函数调用图

Vode 可以记录：

1. 调用了哪些函数
2. 谁调用了谁
3. 调用发生在什么文件、哪一行、什么函数内部
4. 每个函数调用的父调用节点是谁

这意味着可以构建一棵调用树，或者进一步加上数据流边，构成“函数调用 Operation 宇宙”。

---

## 2.3 函数边界上的输入输出映射

对于每次函数调用，可以较稳定地记录：

1. 形参名
2. 实际传入对象
3. 返回对象
4. 参数和返回值的基础元信息

因此可以表示为：

```text
Input -> Function Call -> Output
```

并建立如下关系：

- 某函数返回了对象 A
- 后续哪个函数把对象 A 作为参数传入
- A 因此从前一个函数流向后一个函数

这正是当前方案的核心价值。

---

## 2.4 函数边界上的数据流展示

当前方案可以较好展示：

1. 对象由哪个函数返回
2. 对象之后被哪些函数消费
3. 一个对象是否被多个后续函数共享使用
4. 某个调用节点的输入来自哪些上游调用
5. 某个调用节点的输出流向哪些下游调用

这部分是“来源与流向”的第一阶段版本：**函数级来源**。

---

## 2.5 递归展开函数调用

由于每个调用事件都在调用栈中发生，因此可以记录：

1. 父调用
2. 子调用
3. 子调用的输入输出

所以可以在 UI 中做到：

1. 默认折叠某个函数
2. 点击后展开其内部调用的子函数
3. 只展开单个函数，而不展开兄弟函数

这与原始设想中的“选择性展开”高度一致，但粒度限定为函数调用边界。

---

## 2.6 `torch.nn` 的特殊支持

这一部分是当前方案的重要增强点。

对于 PyTorch / `torch.nn` 场景，可以参考 [`torchview`](../../torchview/README.md) 的思路，对下列对象提供特殊支持：

1. `nn.Module`
2. `named_parameters()` 返回的参数
3. `named_buffers()` 返回的 buffer
4. tensor 元信息
5. module path，例如 `encoder.layer.0.attention`

这样可以让函数调用图在神经网络场景下具备更高解释性，例如：

- 某次调用其实是 `Linear.forward`
- 输入是 activation
- 消费了哪些 parameter
- 返回 tensor 的 shape 是多少

这意味着当前方案虽然主图是函数调用图，但在 `torch.nn` 场景中，仍然能表现出较强的“模型结构理解能力”。

---

## 2.7 变量元信息与受控值显示

当前方案可以对输入输出对象显示：

1. shape
2. dtype
3. device
4. requires_grad
5. numel
6. 统计值
   - min
   - max
   - mean
   - std
7. 前若干个值的 preview

这里必须强调：

- **不会默认存完整 tensor**
- 大 tensor 只记录统计值和截断预览
- 小 tensor 才展示少量样本值

这可以兼顾信息价值与存储成本。

---

## 2.8 静态图与 Web 交互展示

当前方案完全支持：

1. 静态 SVG 图
2. Web 交互式查看器
3. 点击节点展开子调用
4. 点击变量查看详情
5. 搜索函数名、模块名、路径
6. 开关显示 shape、类型、统计值摘要

---

## 3. 当前方案不能实现的功能

## 3.1 不能完整恢复函数内部每一步局部变量流动

例如：

```python
def forward(x):
    h = self.fc1(x)
    h = torch.relu(h)
    y = self.fc2(h)
    return y
```

当前方案可以较好记录：

- `forward(x) -> y`
- `forward` 内部调用了哪些函数
- 各函数之间的参数/返回值关系

但不能稳定恢复：

- 第一次 `h` 和第二次 `h` 的所有表达式级来源链
- `x -> h -> h -> y` 的每一个局部变量版本演化

也就是说，当前方案对“函数内部局部变量级数据流”的恢复能力有限。

---

## 3.2 不能把任意表达式都变成稳定节点

以下内容通常无法自然成为高质量节点：

1. `a + b`
2. `x @ w`
3. 下标访问
4. 属性访问
5. 列表推导式内部中间值
6. 原地修改内部状态

因为当前方案的主节点是“函数调用事件”，不是“表达式执行事件”。

---

## 3.3 不能默认向下展开到底层算子实现

当前方案不追求把：

- `torch.matmul`
- `torch.add`
- `relu`
- 甚至更底层的 ATen / C++ / CUDA kernel

继续拆成更细的内部实现图。

这既不是当前阶段必须的，也不现实。

如果后续真的需要，只能通过：

1. 少量白名单算子专项建模
2. AST/字节码插桩
3. 或手写 decomposition 规则

来增强，而不是自动得到。

---

## 3.4 不能稳定恢复跨作用域的精确变量名链

当前方案可以较好读取：

1. 函数参数名
2. 部分 frame 中的局部变量名
3. 模块参数名与 buffer 名

但不能保证：

1. 同一对象在不同作用域中的唯一主名
2. 历史变量名演化链完整保真
3. 中间表达式临时值的稳定名称

因此“变量名”只能作为展示辅助信息，不能作为系统主键或强语义承诺。

---

## 3.5 不能默认展示所有对象的完整值

对于输入输出对象，只能采用分级策略：

1. tensor：统计值 + preview
2. 小型标量/容器：摘要显示
3. 不可安全序列化对象：类型名 + 简述

所以当前方案强调的是：

- **可用的详细信息**
- 而不是**完整原值转储**

---

## 4. 哪些需求是部分实现而非完全实现

以下能力属于“有一定支持，但不能按理想状态承诺”的范围：

### 4.1 来源追踪

可做到：

- 函数级来源
- 对象是由哪个调用返回的
- 后续被哪些调用消费

做不到：

- 表达式级来源
- 行内子表达式级来源
- 原地修改前后的精细版本图

### 4.2 混合粒度视图

可做到：

- 函数调用主视图
- `torch.nn` 相关对象增强显示

做不到：

- 模块、函数、算子、表达式全部无缝统一为同一种高保真节点体系

### 4.3 高保真数据流

当前方案已经是高保真数据流的**函数边界版本**。

也就是说：

- 对函数输入输出的流向，尽量保真
- 对函数内部表达式层级流动，暂不做强承诺

---

## 5. 一个具体例子

下面用一个简化例子说明当前方案到底能看到什么。

### 5.1 示例代码

```python
import torch
import torch.nn as nn

class SimpleNet(nn.Module):
    def __init__(self):
        super().__init__()
        self.fc1 = nn.Linear(10, 5)
        self.fc2 = nn.Linear(5, 2)

    def forward(self, x):
        h = self.fc1(x)
        h = torch.relu(h)
        y = self.fc2(h)
        return y

model = SimpleNet()
input_data = torch.randn(3, 10)
output = model(input_data)
```

### 5.2 当前方案能稳定得到什么

在函数调用图层面，可以得到类似结构：

```text
Call: SimpleNet.forward
  input:
    x -> Tensor shape=[3, 10]
  children:
    Call: Linear.forward
      input:
        input -> Tensor shape=[3, 10]
      output:
        return -> Tensor shape=[3, 5]

    Call: torch.relu
      input:
        input -> Tensor shape=[3, 5]
      output:
        return -> Tensor shape=[3, 5]

    Call: Linear.forward
      input:
        input -> Tensor shape=[3, 5]
      output:
        return -> Tensor shape=[3, 2]

  output:
    return -> Tensor shape=[3, 2]
```

并且可以建立数据流：

```text
input_data
  -> SimpleNet.forward.x
  -> Linear.forward.input
  -> Linear.forward.return
  -> torch.relu.input
  -> torch.relu.return
  -> Linear.forward.input
  -> Linear.forward.return
  -> SimpleNet.forward.return
  -> output
```

这里的数据流是“函数边界保真”的。

### 5.3 当前方案看不到什么

它很难自动恢复出如下这种更细的语义图：

```text
x -> fc1 -> h1 -> relu -> h2 -> fc2 -> y
```

原因是：

- `h1` 和 `h2` 都是函数内部局部变量
- 方案 A 不以表达式级赋值为主事件
- 它主要记录的是 call / return 边界

因此在 UI 中更现实的展示方式是：

- 以 `Linear.forward`、`torch.relu`、`Linear.forward` 为主节点
- 输入输出对象作为边界变量节点
- 局部变量名 `h` 仅作为辅助标签，不能作为绝对可靠的数据流锚点

---

## 6. 结论

当前方案是一个**收缩后的高可行性方案**。

它最擅长解决的问题是：

1. 程序调用了哪些函数
2. 函数之间的数据如何流动
3. `torch.nn` 场景下哪些 module、parameter、buffer 参与了调用
4. 输入输出 tensor 的元信息和统计值是什么
5. 用户如何通过展开调用节点逐步理解执行流程

它暂时不解决的问题是：

1. 函数内部表达式级数据流
2. 所有局部变量的高保真来源恢复
3. 底层算子内部实现展开
4. 完整 tensor 值转储

所以它并不是“万能执行图系统”，而是：

> 一个以函数调用边界为核心、对 `torch.nn` 进行专项增强、并尽量保证函数级数据流保真的可视化方案。
